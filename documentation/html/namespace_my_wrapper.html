Ancestor.

This include file contains the parent class definitions. Query definitions.

This include file contains the definitions of the query class. Local definitions.

This include file contains all local definitions to this class. Data wrapper.

This class overloads its ancestor to implement a web service that wraps a data store, it represents a framework for building concrete data store web-service wrappers.

The class introduces a series of new operations and filter options that must be implemented in derived classes which implement a specific data store.

These new functionalities require a new set of parameters:

  • Data store parameters: In order to refer to a specific data store we need two parameters:
    • kAPI_DATABASE: Database, this parameter should indicate the database or equivalent concept where the data should be stored or retrieved. This parameter can be compared to the database part of an SQL table reference (DATABASE.TABLE).
    • kAPI_CONTAINER: Container, this parameter should indicate which container within the database should be used to store or retrieve the data. This parameter can be compared to the table part of an SQL table reference (DATABASE.TABLE).
  • Paging parameters: Query results may possibly return large amounts of data, this means that a paging mechanism should be set in place:
    • kAPI_PAGE_START: Page start, this parameter indicates the starting page or record.
    • kAPI_PAGE_LIMIT: Page count, this parameter indicates the maximum number of pages or records that the operation should return.
  • Data parameters: A query is formed by a series of sections, each of which is provided with the following parameters:
    • kAPI_DATA_QUERY: Query, this parameter is used when retrieving data, it represents the filter or selection query; it must be expressed as one of the query class siblings. This parameter can be compared to the WHERE part of an SQL query.
    • kAPI_DATA_FIELD: Fields, this parameter indicates which elements of the selected objects we want returned. This parameter can be compared to the SELECT part of an SQL query.
    • kAPI_DATA_SORT: Sort fields, this parameter indicates the sort order, the list of data elements by which the result is to be sorted. This parameter can be compared to the field parameter or to the ORDER BY part of an SQL query.
    • kAPI_DATA_OBJECT: Object data, this parameter represents the data to be stored in the database, the type should ideally be abstracted from the data store engine. This parameter can be compared to the VALUES or SET part of an SQL query.
    • kAPI_DATA_OPTIONS: Options, this parameter represents the options governing data store and retrieve operations. In general it will cover the options when storing data and the actual implementation is the responsibility of derived classes:
      • kAPI_OPT_SAFE: Safe commit option, this is relevant only when committing data. If this option is OFF, it means we want to perform an asynchronous operation: the store operation will occur in the background and the program execution will not wait for it to finish; this also means that the client is responsible for checking whether the operation completed. If the option is ON, the operation is synchronous, which means that the program will wait for the store operation to complete.
      • kAPI_OPT_FSYNC: File sync option, this tag is relevant only when committing data. If the option is ON, it means that the store operation will wait until the data is actually written to disk; which may not necessarily be the case even if the kAPI_OPT_SAFE option was on. When this option is set, it is implied that the kAPI_OPT_SAFE option is also on. If the option is OFF, it means that the data will be synched to disk only when the buffer is flushed.
      • kAPI_OPT_TIMEOUT: Operation timeout, it represents the time in milliseconds beyond which the client will stop waiting for a response and expect a time out status.
      • kAPI_OPT_SINGLE: First element, this option is used by the delete operation: if ON, only the first object satisfying the query will be deleted; if OFF all selected elements will be deleted.
      • kAPI_OPT_SINGLE: First element, this option

The new operations declared in this class are:

  • kAPI_OP_COUNT: This operation requests a count, which is an integer indicating the total number of elements satisfying the provided query. This number is not to be confused with the page element count described further.
  • kAPI_OP_MATCH: This operation is equivalent to a read query, except that it will try to match one query clause at the time and will return a result on the first match.
  • kAPI_OP_GET: This operation is equivalent to a read query, it requests a list of objects satisfying the provided query.
  • kAPI_OP_SET: This operation is equivalent to an insert for new objects or an update for existing objects, the operation will replace the object in the data store with the one provided in the kAPI_DATA_OBJECT parameter.
  • kAPI_OP_UPDATE: This operation is equivalent to an update operation, this implies that the object must already exist in the data store and that the operation will replace the object in the data store with the one provided in the kAPI_DATA_OBJECT parameter.
  • kAPI_OP_INSERT: This operation is equivalent to an insert operation, this implies that the object must not already exist in the data store.
  • kAPI_OP_BATCH_INSERT: This operation is equivalent to the kAPI_OP_INSERT operation, except that we provide here a list of objects to be inserted.
  • kAPI_OP_MODIFY: This operation indicates that we want to modify the contents of an existing object and that the provided data represents only the changed elements.
  • kAPI_OP_DEL: This operation indicates that we want to delete the elements matching the provided query: the first one only, if the provided kAPI_OPT_SINGLE option is on, or all if off or omitted.

The added functionality implies that a series of additional sections will be returned in the response:

  • kAPI_DATA_PAGING: Paging section, this section will return the paging information of the current operation, besides the provided start and limit parameters, it will also feature:
    • kAPI_PAGE_COUNT: This element will hold the actual number of returned objects, this number will be either equal or smaller than the provided limit parameter.

This class implements a graph node.

The class is derived from CPersistentObject, of which it inherits the persistence framework, but it differs profoundly in that, although the class is an ArrayObject, the actual array that it manages is the properties array of the node. In other words, when you add, retrieve and delete properties, you are not doing so with the internal array, but with the node's properties array.

The class features a single data member, the node which contains a Neo4j node reference.

Since the object requires a valid node to exist at all times, this may either be an empty one, if we wish to create a new node, or a loaded one, this means that it is not possible to use the inherited interface of instantiating the object with array contents, the container provided to the constructor must be a Neo4j client.

Note that the class will not cast to an array correctly, you must use the getArrayCopy method to get an array, if you know how to solve this, please do it!

 @subpackage        Wrappers

 @author            Milko A. Škofič <m.skofic@cgiar.org>
 @version   1.00 03/06/2011
                    2.00 22/02/2012

Ancestor.

This include file contains the parent class definitions. Types.

This include file contains all data type definitions. Local definitions.

This include file contains all local definitions to this class. Wrapper.

This class represents a web-services wrapper server, it is the ancestor of web-service classes in this library.

Wrappers are objects that respond to standard web calls and receive parameters in both GET and POST parameters, the main two parameters handled by this class are:

  • kAPI_FORMAT (required): The data format of the response, it will be returned as:
    • kTYPE_PHP: A PHP-serialised string.
    • kTYPE_JSON: A JSON-serialised string.
  • kAPI_OPERATION (required): The requested operation, each class will implement specialised handlers, this class only implements the following two operations:
    • kAPI_OP_HELP: A LIST-OP command, this command will return in the response section the list of supported operations as an array structured as follows:
      • index: The index will be the operation code.
      • value: The value will be the operation description.
    • kAPI_OP_PING: A PING command, this can be used to check if a service is alive.

If both the above parameters are present, the service will return an array constituted by the following three sections:

  • kAPI_DATA_STATUS: Operation status. This section is returned by default and will inform on the status of the requested operation. It consists of an array containing the following elements:
    • kTAG_STATUS: Response status. This element will be returned by default regardless of the operation outcome. This corresponds to the severity of the response and it can take the following values:
      • kMESSAGE_TYPE_IDLE: This is the status of the web-service before any operation has been executed, or when the operation was successful; this is the response of a successful ping request.
      • kMESSAGE_TYPE_NOTICE: The operation was successful and a notice message was returned.
      • kMESSAGE_TYPE_MESSAGE: The operation was successful and a message was returned.
      • kMESSAGE_TYPE_WARNING: The operation was successful but a warning was raised.
      • kMESSAGE_TYPE_ERROR: The operation failed because of an error.
      • kMESSAGE_TYPE_FATAL: The operation failed because of a fatal error, this will generally mean that the web-service is not operational.
      • kMESSAGE_TYPE_BUG: The operation failed because of a bug, the developers should be informed of this kind of errors.
    • kTAG_CODE: Status code. This element will be returned by default regardless of the operation outcome. It corresponds to the error code; zero means no error.
    • kTAG_DESCRIPTION: Status message. The response message from the operation, this element is used to return informative messages or to return error messages when the service fails. It will generally be formatted as an array structured as follows:
      • kTAG_TYPE: The data type of the message, it will be a string in general.
      • kTAG_LANGUAGE: The language ISO 639 2 character code in which the message is expressed in.
      • kTAG_DATA: The actual message data contents.
    • kAPI_AFFECTED_COUNT: Record count. The total number of elements affected by the operation. This tag will only be used by derived classes returning data elements.
    • kTAG_ANNOTATION: Attachments. A list of key/value pairs containing information relevant to the operation response. For instance, if a series of parameters are required and were not provided, this could list them.
  • kAPI_DATA_REQUEST: Service request. This section will return the actual request provided to the service, this can be used for debugging purposes and will only occur if the optional kAPI_OPT_LOG_REQUEST parameter has been set to 1.
  • kAPI_DATA_TIMING: Timers. This section holds timing information, it will be returned only if you provide the time of day [gettimeofday( TRUE )] in the kAPI_REQ_STAMP parameter. This section is structured as follows:
    • kAPI_REQ_STAMP: Request time stamp, the time in which the request was sent; this is the same value sent by the caller in the kAPI_REQ_STAMP parameter.
    • kAPI_PARSE_STAMP: Parse time stamp, the time in which the service finished parsing the request.
    • kAPI_RES_STAMP: Response time stamp, the time in which the response was sent.
  • kAPI_DATA_RESPONSE: Response, this section will hold the operation response, in this class we only respond to operations list requests.

Besides the format and operation parameters described in the first section, we have three other optional parameters that can be used to receive specific information sections in the response:

  • kAPI_OPT_LOG_REQUEST: Log the request, if the value of this parameter is 1, the response will contain the received request in the request section.
  • kAPI_OPT_LOG_TRACE: Trace exceptions, if the value of this parameter is 1, in the case of an error that triggered an exception, the error response will also include the call trace.
  • kAPI_REQ_STAMP: This parameter should hold the timestamp [gettimeofday( TRUE )] in which the client has sent the request, if provided, the service will return the timing information in the timing section.

The parameters are expected either in GET or POST.

If either the format or the operation parameters are omitted from the request, the service will return an empty response; this is to prevent unnecessary traffic.

Instances of this class can be considered server objects, and can be implemented with this simple code snippet:

$server = new CWrapper(); $server->HandleRequest();

An example of this class implementation can be found in the Wrapper.php source file.
