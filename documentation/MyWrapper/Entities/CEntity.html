Entity.

An entity can be an individual, and organisation or a legal entity that must be referenced.

The main difference between objects derived from this class and other instances of people or organisations is that in the latter case these entities may be embedded in in other objects, whereas in the case of this class, these entities must exist on their own since they are to be referenced by different sources.

In this class we declare only the bare minimum attributes that any entity needs, derived classes will use this one as a base to implement concrete instances:

The shared attributes are:

  • kTAG_CODE: This attribute is a short string that can be used to discriminate the entity, the string should be readable and printable. In general, this string should be unique among all entities of the same type, although this may not be required as long as duplicates are not mixed. The class features a member accessor method to manage this property.
  • kTAG_TYPE: This offset represents the entity type. This attribute should not be confused with the object's class: the latter provides an indication on the functionality and structure of the entity, whereas this attribute provides an indication on the nature of the entity. This attribute is implemented as an array. The class features a member accessor method to manage this property.
  • kTAG_NAME: This offset represents the entity name. It may be considered an expanded version of the code or a label that can be applied to the entity. By default it should be a string, concrete derived instances may expand on this. The class features a member accessor method to manage this property.
  • kOFFSET_EMAIL: This offset represents the entity e-mail address. It is a single element and should be always up to date.
  • kTAG_REFS: This offset represents the list of references of the current object, it is implemented as an array whose elements are structured as follows:
    • kTAG_TYPE: This offset represents the reference type or context.
    • kTAG_DATA: This offset represents the reference itself, it is the following structure:
      • kTAG_ID_REFERENCE: The unique identifier of the referenced object.
      • kTAG_CONTAINER_REFERENCE: The container name.
      • kTAG_DATABASE_REFERENCE: The database name.
      • kTAG_CLASS: The object class name.
    The class features a member accessor method to manage this property.

Objects of this class require at least the code offset to be set if they expect to have an initialised status.

When committing, eventual reference elements set as the actual instances will be first saved to the same container, then replaced by references.

The class also features a static method that returns the default container name for objects of this type.

Affiliate (line 393)

Manage entity references.

An entity may reference a series of other entities and each reference may have a different type. This method can be used to manage the entity offset used to store these relations.

The method accepts the following parameters:

  • $theType: Type of reference, this is a scalar, possibly a string, that indicates the type of reference we are adding, retrieving or deleting. This value may be NULL for references that do not imply a type.
  • $theValue: Reference or object. This parameter represents the reference, it may be a scalar, representing either the referenced object identifier, a structure representing an object reference, or the referenced object itself.
  • $theOperation: The operation to perform:
    • NULL: Return the element matched by the previous parameters.
    • FALSE: Delete the element matched by the previous parameters and return it.
    • other: Any other value means that we want to add to the list the element provided in the previous parameters, either appending it if there was no matching element, or by replacing a matching element. The method will return either the replaced element or the new one.
  • $getOld: Determines what the method will return when deleting or replacing:
    • TRUE: Return the deleted or replaced element.
    • FALSE: Return the replacing element or NULL when deleting.

Each element of this list is structured as follows:

  • kTAG_TYPE: This offset represents the reference type or class, it may be omitted if the reference has no type or when we want to define a default reference. The first parameter will be stored here.
  • kTAG_DATA: This offset represents the reference itself, it may be in two forms:
    • Scalar: A scalar is interpreted as the object's identifier.
    • Reference: An object reference structured as follows:
      • kTAG_ID_REFERENCE: The unique identifier of the referenced object. This element is required by default.
      • kTAG_CONTAINER_REFERENCE: The container name. This element is optional.
      • kTAG_DATABASE_REFERENCE: The database name. This element is optional.
    • Object: An object derived from this class will be interpreted as the referenced object itself. When committing the current object, these objects will also be committed and converted to references.
